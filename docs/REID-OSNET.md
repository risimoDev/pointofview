# OSNet ONNX для сквозной идентификации (Re-ID)

## Зачем

По умолчанию Re-ID сравнивает людей по HSV-гистограмме одежды. Это работает
днём при цветной картинке, но **ночью в ИК-режиме камера даёт почти чёрно-белое
изображение** — цветовые гистограммы превращаются в шум, и один и тот же человек
при каждом появлении выглядит «новым» (отсюда 44 посетителя вместо 2 курьеров).

OSNet — нейросеть, обученная именно для re-identification людей: она сравнивает
силуэт, текстуру, пропорции, а не только цвет, и устойчива к ИК/ночной съёмке.
**Обучать ничего не нужно** — берётся готовая предобученная модель.
Файл интегрируется одной переменной окружения `REID_ONNX`; при её отсутствии
анализатор автоматически работает на гистограммах.

## Шаг 1. Получить ONNX-файл (на dev-ПК, нужен интернет)

```bash
pip install torch torchvision gdown
pip install git+https://github.com/KaiyangZhou/deep-person-reid.git
python scripts/export_osnet.py
```

Ставить torch на прод-сервер не нужно. Если под рукой только сервер — экспорт
делается в одноразовом контейнере, ничего не оседает в системе:

```bash
cd ~/pointofview
docker run --rm -v "$PWD:/w" -w /w python:3.12 bash -c "
  pip install -q torch torchvision gdown &&
  pip install -q git+https://github.com/KaiyangZhou/deep-person-reid.git &&
  python scripts/export_osnet.py"
```

Файл `osnet_x0_25.onnx` появится в корне репозитория (≈2.5 ГБ качается один раз,
занимает время; сам файл модели — около 9 МБ).

Скрипт качает готовые веса `osnet_x0_25` (самая лёгкая, CPU-инференс ~5 мс на
кроп), экспортирует `osnet_x0_25.onnx` и печатает размерность эмбеддинга.
Обучать ничего не нужно.

## Шаг 2. Положить модель на сервер

```bash
# с dev-ПК (путь DATA_ROOT смотри в infra/.env.prod на сервере, обычно /opt/viziai/data)
scp osnet_x0_25.onnx user@viziai-server:/opt/viziai/data/models/
```

Каталог `${DATA_ROOT}/models` монтируется в контейнер анализатора как `/models`
(read-only) — это уже прописано в `infra/docker-compose.prod.yml`.

## Шаг 3. Включить на сервере

В `infra/.env.prod` добавить:

```
REID_ONNX=/models/osnet_x0_25.onnx
```

Перезапустить анализатор:

```bash
docker compose -f infra/docker-compose.prod.yml --env-file infra/.env.prod up -d --build analyzer
```

## Шаг 4. Перенастроить пороги (ОБЯЗАТЕЛЬНО)

У OSNet другая шкала косинусной близости. В админке `/admin/features` →
«Сквозная идентификация» поставить стартовые значения:

| Параметр | Гистограммы (старое) | OSNet (новое) |
|---|---|---|
| Порог совпадения | 0.88 | **0.70** |
| Порог сотрудника | 0.90 | **0.75** |

Дальше подстраивать по месту: если один человек всё ещё дробится на несколько —
порог совпадения понизить (0.65); если разных людей склеивает — повысить (0.75).

## Шаг 5. Сбросить обучение и заново отметить сотрудников

Эмбеддинги гистограмм (384 числа) и OSNet (512 чисел) несовместимы: после
перехода старая галерея и отметки сотрудников автоматически игнорируются
(анализатор пропускает записи с неподходящей размерностью), но продолжают
висеть в списке. На странице «Люди» внизу есть блок **«Сброс обучения»** —
нажать «Сбросить всё, включая сотрудников», затем:

1. добавить сотрудников по имени (кнопка «Добавить сотрудника»);
2. загрузить каждому 1–3 чётких фото лица анфас;
3. когда сотрудник появится на камере — отметить его образ кнопкой «+»
   в его карточку (это даёт эталон одежды).

Если один человек снова размножился на несколько карточек — отметить их
галочками и нажать «Объединить»: образцы сложатся в одну карточку.

## Проверка

1. `docker compose ... logs analyzer | grep -i reid` — не должно быть ошибок ONNX.
2. Страница «Люди»: за день число «замеченных» должно примерно соответствовать
   реальному потоку, без десятков дублей одного человека.
3. Ночной тест: курьер зашёл/вышел несколько раз → одна личность, а не несколько.
